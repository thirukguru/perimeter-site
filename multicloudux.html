Scope — where it applies
Grants attach to any level of the cloud hierarchy and inherit downward:
All clouds
 └ Connection (Entra tenant / AWS org)
   └ Scope (mgmt group / folder)
     └ Target (account / subscription / project)
Same tree the scanner uses — one hierarchy, not a parallel permissions model.
Numbers follow the grant, not the tenant. Jane sees a compliance score and coverage computed over AWS Production only — not the tenant-wide 78%. A global figure shown to a scoped user leaks how other teams are doing. Scoped access without scoped aggregation is an information leak wearing an access-control costume.
Why role × scope, and not more roles The instinct is to add roles ("Azure admin", "AWS auditor") — that multiplies with every cloud you support. Keeping four roles and making scope the second axis means adding GCP adds zero roles. Owner and Admin stay unscoped on purpose: granting directory access or rotating a credential is a tenant-wide trust decision, so it can't sit behind a per-account grant.

Kill chain
1Initial access. App Gateway publishes the VM on :443 — route table and NSG both permit it.
2Execution. Unpatched CVE-2024-23652 on the VM allows remote code execution.
3Credential access. VM carries a managed identity with a federated credential.
4Lateral movement — across clouds. The identity satisfies the AWS role's trust policy via AssumeRoleWithWebIdentity.
5Collection. The role grants s3:GetObject on a bucket tagged customer-data.
Break the chain
1Scope the AWS trust policy to a single subject claim cheapest cut
2Patch CVE-2024-23652 on the VM
3Remove s3:GetObject * from the role
Cutting step 4 breaks this path and 2 others that traverse the same trust policy.
The differentiator isn't the picture — it's the edge Anyone can draw a graph. The question is whether an edge is a fact or a guess. Ours are labelled: proven edges are computed from route tables, NSG/SG rules and trust policies; inferred edges are correlated findings where reachability was never established — and an unassessed area appears as its own path state rather than a missing line. A drawn edge is a claim about reachability; if we can't prove it, we say so. The path above crosses the federation boundary — Azure identity into AWS data — which is the risk class that appears the moment a customer runs two clouds, and which single-cloud tools structurally cannot see.

Narrowing scope is safe here — and only here. Whatever you exclude is recorded as not assessed and subtracted from your coverage percentage. It is never counted as a pass, and findings in excluded regions are never auto-resolved.
Why scope and schedule are separate pages Scope answers "what do we assess" and directly determines your maximum achievable coverage — it's a security decision. Schedule answers "how often", which is an operations decision about cost and freshness. Bundling them hides the fact that narrowing scope silently caps the coverage number. Two more notes: the per-scan override lives on Run a scan, and Azure has no region list — locations are resource properties there, so we follow the delegation instead.

What happened

The storage account stcontosoprod01 permits anonymous read access at the container level. Any unauthenticated caller can enumerate and download blobs. Azure evaluates public access at both the account and container level — this account allows it at the account level, so per-container settings cannot fully contain the exposure.
